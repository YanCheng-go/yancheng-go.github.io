Agents, humans, and the harness

"Walk through this diagram in four clicks. First click — the harness. Two halves: the repo as single source of truth — AGENTS.md, architecture, conventions, specs. And tooling for self QA — DevTools, logs, metrics, linters, traces. This is the environment agents live in. Second click — what agents produce. Product code, tests, CI/CD, documentation, reviews. They open PRs, self-review, do agent-to-agent review, iterate, merge. Plus garbage collection — periodic background scans for drift. Third click — the self-fix loop and humans. When agents struggle, they first try to self-fix — that green loop going back up. On the left, humans proactively: design systems, define intent, prioritize work. Both feed into the harness. Fourth click — diagnose gaps. When self-fix fails, it escalates to humans. The question: what's missing — a tool, a guardrail, a doc? Fix that, and it feeds back up to improve the harness. Every gap closed makes all future work better. The key insight: humans never write code. They build and improve the harness. The harness makes agents better. Agents generate everything."

"When the agent struggled, they didn't try harder — they encoded what was missing."

"They didn't get it right on day one. First: massive AGENTS.md — over a thousand lines. Agents pattern-matched locally instead of navigating intentionally. Fix: a ~100 line table of contents. Agents discover details on-demand. Second: every Friday, manual cleanup. Ate 20% of their time. Fix: automated garbage collection — agents periodically scan for drift and open fix PRs themselves. Third: knowledge scattered across Confluence, Teams, someone's head. Invisible to agents at runtime. Fix: everything versioned in the repo. If it isn't discoverable in the repo, it doesn't exist for the agent. The pattern: every time something broke, they encoded what was missing. That's the key mindset shift."

"So what changes for engineers? Four new skills emerge. Environment Design — structuring repos and tooling so agents can navigate effectively. Prompt Decomposition — breaking large goals into agent-tractable building blocks. Feedback Loops — linters that teach fixes, tests that enforce architecture mechanically. And Gap Diagnosis — when an agent fails, the question isn't 'how do I do this myself?' but 'what's missing that would let the agent succeed?' These aren't entirely new skills — they're evolutions of what good engineers already do. The shift is in where you apply them."

"How do we actually get there? For engineers: it's a mindset shift toward more architectural thinking. Start small — cleanup, tests, docs. Invest in learning and sharing. Our debugging skills transfer directly — diagnosing agent gaps is the same diagnostic muscle. From leadership: celebrate environment wins like feature launches. Treat tooling time as product time. Expect a slow start — that's investment, not failure. And track how each improvement reduces future failures — that's how you make the compounding effect visible."

Pick a low-stakes pilot

Internal tooling, test generation, doc updates

Treat every agent failure as a system problem

Don't blame the model — fix the environment

"Three things at the organizational level. One: bring knowledge into the repo. If architecture decisions live in Confluence pages, conventions are in someone's head, specs are scattered across Teams channels — agents can't use any of it. Move it into the repo. That's the single biggest unlock. Two: pick a low-stakes pilot. Internal tooling, test generation, doc updates. Not the critical production system. Give the team space to learn. Three — the mindset shift: treat every agent failure as a system problem. When the agent produces bad output, ask: what was missing in the environment? Fix that. Now every future task benefits. That's harness engineering."
